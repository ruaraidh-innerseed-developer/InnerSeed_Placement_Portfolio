01

Step 1 — Close out Preset mode

1

Carry-over from Week 2 — quick, should be resolved early in the week

Two items remain open from the Week 2 Manifesto's running note
Confirm the Preset session completion flow

What actually happens when a Preset session ends — does it return to the dashboard, the player, or somewhere else? This connects directly to the Week 1 "post-session landing screen" idea (W1-04) and to D-13 (no way back from the player). Confirming this closes the loop on the full Preset journey, start to finish.

Filter mode — tested within the Preset browsing flow

Rather than a separate pass, Filter mode gets tested as part of browsing for a session in Preset mode — is it discoverable, does it behave as expected, and does it interact cleanly with the Load Session modal flagged in D-19/D-20.

Why this comes first

This is the last unfinished piece of the Preset-mode picture. Closing it out means Week 3's main event — Builder mode — starts from a complete, comparable baseline.

02

Step 2 — Builder mode: full UX audit

2

Headline focus of Week 3

Mirrors the structure used for Preset mode in Week 2
Live build-through

Build a session from scratch in Builder mode, exactly as a new user would, narrating the experience as it happens — entry point, options, controls, and the save flow at the end. This is the direct counterpart to the "Ethereal Voyager" Preset walkthrough from Week 2.

Same lens, same questions

Apply the same structured approach that produced D-01 to D-21: clarity, orientation, friction, and — critically — does Builder mode show the same "three disconnected navigation systems" pattern (D-21), or does it handle saving, loading and favouriting differently? Each finding gets logged in the same D-xx format, continuing the running log.

Use Claude for visuals and suggestions

Wherever a friction point is found, Claude produces a quick mockup or annotated wireframe in the same style as the Week 2 "threefold consistency" panels — showing current state alongside a proposed alternative. Where a finding connects back to D-21, the mockup should show how the existing threefold structure would (or wouldn't) extend cleanly into Builder mode.

What this tells us

By the end of this step, Week 2's headline recommendation gets its first real test: does threefold consistency hold up across two modes, or does Builder mode reveal something the Preset-only view couldn't show?

03

Step 3 — Mobile testing, all fronts

3

Stress-test, not a parallel audit

Runs across both Preset and Builder mode, now both audited on desktop
Re-run the core journeys on mobile

Both the Preset journey (Ethereal Voyager) and the new Builder journey from Step 2 get repeated on a phone. The goal isn't to find a fresh set of unrelated mobile bugs — it's to see whether the issues already logged (especially D-04 hover-only nav, D-12 tiny tap targets, and D-21's three disconnected navigation systems) become more or less severe on a small touch screen.

Particular attention — D-04 and D-21 on touch

D-04 (left-hand nav only appears on hover) is a desktop-specific problem in its current form — hover doesn't really exist on mobile, so this finding needs re-stating for a touch context. Likewise, three separate navigation systems (D-21) that are merely confusing on a large screen may become genuinely unusable on a small one. Both outcomes — "worse on mobile" or "different on mobile" — are useful evidence either way.

Output

A short mobile findings set (continuing the D-xx log), framed explicitly as evidence for or against the threefold consistency proposal — not as a standalone mobile report.

04

Step 4 — Analytics and human testing: groundwork

4

Delivered in Week 4, planned now

Both now have a hypothesis, thanks to Steps 1–3
Human testing brief

With both Preset and Builder mode audited on desktop and mobile, the brief for human testing becomes specific: can a first-time user find their favourites, complete a session, build one from scratch, and find their way back — and does the threefold consistency structure (if mocked up by this point) make that noticeably easier? Ruaraidh to arrange participants and logistics; Claude to help draft the testing script and the specific tasks/questions once Steps 1–3 are complete, so the brief reflects the final state of the findings log.

Analytics framework — what to measure, not how to build it

A short, evidence-linked list of what would confirm or rule out the Week 2–3 findings if it were tracked: do users reach the favourites they've saved (D-18), do they get stuck or scroll past the Load Session modal (D-19), do they end up in the wrong mode after an action (D-11/D-20). This stays a recommendations list for Jeroen — the technical implementation is his call.

External UX professional

Their informal review is also expected in this window. Cross-referencing their feedback against the D-01 to D-21+ log (and against the human testing results, once available) is a natural fit for Week 4.
