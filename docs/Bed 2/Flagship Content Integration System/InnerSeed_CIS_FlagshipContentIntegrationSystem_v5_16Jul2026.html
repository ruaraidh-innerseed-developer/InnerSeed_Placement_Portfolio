Starting point

Zero app awareness, two cities, one Stack

This document treats the app's content system as starting from nothing — no content development system in place, no audience awareness of the app itself — regardless of any personal or social following either founder already has elsewhere. That following is real, but it is not app awareness, and this plan doesn't lean on it.

  • NL · Enschede Jeroen — building app awareness in his home city, extending the breathwork practice already running there.
  • UK · Edinburgh Ruaraidh — building app awareness in Edinburgh from zero, training as a breathwork coach alongside the marketing work, so this route grows into actual local delivery, not just promotion.

Two local, B2C-first motions, run in parallel on one shared Stack, the goal in both places the same: reach people who'd genuinely benefit from this, make them aware the app exists, and move them toward it — directly for Breathers, and as the visible proof that makes the B2B2C and App-Shell conversations credible later.

Ideation layer

Where ongoing content ideas come from

The Message Bank carries the brand's core voice — the throughline every piece should still sound like. It was never going to survive as the only source of what to post this week. The system needs a separate, renewable ideation layer feeding the Insight hat every cycle.

App feature stories

D-43 (Spotify import) is the first. Every feature that ships from here is another, ready-made and true.

Founder and coach practice, behind the scenes

Jeroen's own ongoing practice; Ruaraidh's own coach training and his learning curve building this from zero in Edinburgh. Genuine because neither is performed.

Local taster sessions

Small in-person sessions either founder runs to seed real word of mouth — content and community-building in one act.

Category & seasonal hooks

The recurring wellness-calendar moments the category leans on — a scaffold, never the whole plan.

Early user stories

Empty today by definition, but the first real Breather stories become a source the moment they exist.

Message bank threads

The core voice running through several of the above, not a seventh source competing with them.

ClaudeOperating intelligenceIn place
Suitability profile

The only seat touching every stage of the loop, and the only thing standing between "two markets, one brand" and "two markets, two half-related brands." Mechanically, it runs as an agent loop: Claude reasons about what's needed, calls the relevant tool, receives the result, and continues — the same pattern whether that tool is a Higgsfield generation, a Postiz post, or a MailerLite trigger.

Technical input

Raw material filed in Drive, tagged to its city; standing project context; direct uploads at session start; the ideation sources above.

Technical output

Per pass: insight notes; a strategy plan (audience, route, format, channel, language, market); finished copy plus a structured visual brief object; a quality-checked draft with a pass/hold note.

Interactions — the brief object passed to Higgsfield
{ "city": "edinburgh", // edinburgh | enschede | international "language": "en", // en | nl "route": "b2c", // b2c | b2b2c | app_shell "stage": "awareness", "format": "short_video_9x16", "channel": "instagram", "copy": "...", "visual_brief": { "subject": "...", "mood": "...", "reference_asset": "drive://..." }, "quality_status": "pending" }

Insight and Strategy run per city, never blended. Production and Quality check the object above before it moves. Approved objects route to Higgsfield, then Postiz or MailerLite, each still carrying city, language, and route. Content that is deliberately not city-bound — X, aimed at an international coaching audience — carries international as its market value instead of a city.

HiggsfieldVisual & video productionIn place
Suitability profile

Hosted, multi-model, no local setup — concrete for two founders in two countries with no shared studio. Developer access runs through cloud.higgsfield.ai: REST endpoints plus an official SDK.

Technical input

The visual_brief object from Claude, plus any real reference asset it points to.

Technical output

Generated video/image assets, light-mode native, sized for the destination channel.

Interactions — generation is asynchronous

1. Claude submits the brief as a generation request and receives a job reference back immediately, not a finished asset.

2. Claude (or the orchestration script, if this ever runs outside a chat session) polls the job's status on an interval until it reports complete.

3. The completed asset's URL returns to Claude's Quality hat before either founder sees it. Nothing publishes straight from Higgsfield.

Open technical note: unlike Postiz, no native MCP server is confirmed for Higgsfield — this handoff is a scripted API call today, not a chat-native tool call, until that changes.

Contextual example — either city

A real ten-second screen recording of the Spotify import (D-43) as the reference asset, paired with a brief calling for "the import treated as a small ceremony, not a settings toggle." Output: a 15-second vertical clip, on-brand regardless of which city's operator requested it.

PostizSocial scheduling & distributionSelf-hosting pending Jeroen's confirmation
Suitability profile

Open-source, self-hostable, continuing the owned-over-rented pattern Plausible set. Ships its own CLI and MCP server built specifically so tools like Claude can draft and schedule directly — the single biggest lever for closing "Claude wrote something" to "it went out."

Technical input

Approved copy, approved asset, target integration ID, city tag, scheduling time — only post-approval.

Technical output

Scheduled/published posts; a post ID per post; engagement numbers, readable per city.

Interactions — actual call shape
POST /public/v1/posts { "integration_id": "ig_innerseed_main", "type": "schedule", "content": "{{copy}}", "media": ["{{higgsfield_asset_url}}"], "scheduled_at": "2026-07-21T09:00:00+02:00", "tag": "city:edinburgh,route:b2c" }

Rate limit: 90 create-post calls/hour, self-hosted, configurable — schedule in weekly batches rather than under a deadline.

Contextual example — both cities, one instance

An Edinburgh clip and an Enschede clip schedule separately through the same instance, each tagged, so next week's Insight pass reads two honest, separate answers.

What Claude needs to make this run

Operating requirements, not caveats

1 · Standing context, kept current, for both cities

The project space stays the working memory for Enschede and Edinburgh alike.

2 · Direct tool connections where they genuinely exist

Postiz's MCP server is the priority. Higgsfield's handoff stays a scripted API call until an MCP path is confirmed.

3 · A written quality standard

Brand-voice fit, local-flavour fit, factual accuracy against the live app, fidelity to the message bank, and the standing rule that nothing reads as AI-authored — applied every time.

4 · Real material, continuously fed, from both places

Footage, screenshots, transcripts, voice notes — the difference between content that sounds like the brand and content that sounds like anyone's.

5 · A human-gate protocol, agreed in advance

Work queues and waits if neither founder is available — nothing publishes on a timer alone. The working pattern, confirmed first for X: each channel's owner approves that channel's content — Jeroen for X — with both founders on anything shared. Ownership decides who approves; it never removes the gate itself.

Completion parameters

What keeps input from ending up unfinished

Agreed 17 July: the same five rules apply whichever way content enters — new footage, an existing asset pulled back out of Drive, or a structured feed from another folder. The input changes what Insight has to work with; it doesn't change what happens after.

1 · One pipeline, any input

New footage, an old clip, or a structured feed all enter at Insight and run the same four passes. No input type gets a shortcut or a separate track.

2 · Each pass ends on a condition, not a feeling

Insight ends when it names one specific, usable opportunity. Strategy ends when every field in the brief is filled — audience, route, format, channel, language, and market (a city, or international) — and an unfillable field is logged as an open question attached to the piece, not silently dropped. Production ends at a complete draft, not a rough note. Quality ends at exactly one of two states: pass or hold.

3 · A hold always returns with instructions

Never silence, never a piece that quietly disappears into a backlog. It's revised and re-checked, or explicitly parked with a reason on record.

4 · Claude's finish line is "ready for approval," not "perfect"

That's the actual definition of done from Claude's side — nothing is left open-ended, because there's nothing past that point Claude is meant to be doing alone.

5 · Old content gets the same Insight pass as new content

Nothing skips the pipeline for being pre-existing. An archived clip still needs Insight to say what it's useful for now — otherwise "reuse the archive" quietly becomes "repost with no thought," which ships something unfinished even when it technically goes out.

Failsafe

Six failures, staged and resolved

Each of these is walked through as if it happened this month — the trigger, what the system does next, and where it ends up. None of these are live incidents; they're worked scenarios, so the answer to "what would we actually do" already exists before it's needed.

Scenario 1 · Higgsfield changes its pricing or shuts its API

Trigger: a generation request fails with an account error, or a pricing email arrives making the seat unviable. What happens: production of new visuals pauses — nothing else stops. Copy-only work (X, email, feed captions) continues through Postiz and MailerLite untouched, because no other seat depends on Higgsfield's output to function. Claude's next Insight pass flags the gap explicitly rather than working around it silently. Resolution: the seat is re-tendered — the Flagship already names the principle ("the seat is permanent, the occupant is reviewed"), and the visual-production brief format is tool-agnostic on purpose: the same brief object works for any competent replacement. Worst case, one to two weeks of stills-only content, already survivable at current cadence.

Scenario 2 · A batch hits Postiz's rate limit mid-schedule

Trigger: a scheduling run returns 429 errors partway through a batch. What happens: the remaining posts stay queued locally with their intended times intact — nothing is lost, nothing posts out of order. The retry resumes after the hourly window resets. Resolution: at two cities' current volume this ceiling is far away; the standing practice of batching a week ahead means a one-hour delay in scheduling never becomes a one-hour delay in publishing. If volume ever genuinely approaches 90 posts/hour, the self-hosted config value is raised — it's a setting, not a wall.

Scenario 3 · Quality drift — month three, the voice has gone generic

Trigger: Jeroen or Ruaraidh reads a drafted caption and can't tell it apart from any wellness brand's. What happens: that reaction is itself the signal — it goes in as Insight-pass input, same as any analytics number. The written quality standard is re-read against the last five published pieces, and the specific drift (tone? vocabulary? structure?) is named, not just felt. Resolution: the standard gets a new named check for whatever drifted, and the message bank gets refreshed from new raw material — a founder voice note, a real session moment — because drift usually means the inputs went stale before the outputs did.

Scenario 4 · A three-week stretch where neither founder has hours

Trigger: travel, illness, work pressure — no one available to approve. What happens: the pipeline keeps producing to "ready for approval" and stops there, exactly as the completion parameters define. The queue grows; nothing publishes; the accounts go quiet. Resolution: quiet is the designed outcome, not a failure — a silent fortnight costs almost nothing at this stage, while one unreviewed piece publishing wrongly could cost real trust. When either founder returns, the backlog is already gate-ready in priority order, so the restart is an approval session, not a production sprint.

Scenario 5 · Ideation collapses to one well

Trigger: four consecutive pieces all trace back to message-bank threads alone — no feature stories, no local sessions, no user material. What happens: the Insight pass includes a standing check against the six named sources; a run of four from one source trips it. Resolution: the next cycle is deliberately sourced from whichever wells went unused — and if a well is empty because no raw material has been filed (no new footage, no session recordings), that's raised with the founders as an input problem, named plainly, rather than papered over with more message-bank derivations.

Scenario 6 · Automation outruns oversight

Trigger: a future scripted pipeline (Level 2 or 3 below) is misconfigured so a piece routes toward publishing without a recorded approval. What happens: the interrupt pattern is the defence — the publish step doesn't check whether approval was requested, it halts unless an approval exists. A missing approval means a stopped pipeline, not a published mistake. Resolution: the stopped piece surfaces in the queue with its state intact; the misconfiguration gets fixed; nothing reached an audience. This is why the gate is built as a hard halt rather than a notification — notifications can be missed, halts can't be.

Automation depth

How much of this actually runs itself

This isn't a vague choice any more — it has a concrete technical shape, whichever level gets built.

The mechanism, regardless of how far this goes: Claude runs as an agent loop — it reasons about what's needed, calls a tool (Higgsfield, Postiz, MailerLite), receives the result, and continues, the same pattern at every stage. The publish gate is an interrupt: execution actually pauses before distribution, the pending state is saved, and nothing resumes until a real approval comes back — not a polite suggestion to wait, an actual halt.

Level 1 — Claude in a session, tools connected (lowest build cost). Ruaraidh or Jeroen opens a session; Postiz is reachable directly via its MCP server; Higgsfield via a scripted call. This is close to what's already running.

Level 2 — a lightweight script. A plain Python loop holding the message history, calling each tool's API directly, with the interrupt pattern implemented as a simple pending-approval flag in a database. Full control, but retries, auth, and error logging are all manual. A reference implementation is filed with these documents — InnerSeed_CIS_Level2_PipelineReference_19Jul2026.py — covering both entry paths (new footage and archived asset), a real Claude quality pass that can hold, the interrupt as a pending-approval file that halts execution, and the post-approval Postiz call. Unrun in this environment by design.

Level 3 — a structured workflow tool (n8n or LangGraph). Built-in state persistence and a native interrupt/resume mechanism, at the cost of a steeper setup and, for n8n, a visual workflow to maintain instead of code.

Which level, and who builds and maintains it, is Jeroen's call — it's his infrastructure to run day to day. What this document commits to regardless: the human gate is a real pause, not an assumption, at every level.

Still open

Two decisions this document doesn't make

Exact weekly capacity. The plan assumes a sustainable, repeatable rhythm rather than a first-month sprint — but it still hasn't been given a real number of hours from either city. That number decides whether the starting cadence is one strong piece a week per city, or more.

The paid-ads seat. Part Two names the gap plainly: the five-instrument Stack has no ad-platform seat. Once a budget exists, paid spend runs through Meta Ads Manager or YouTube Ads directly — not Postiz, since scheduling tools and ad platforms are different jobs. Whether a sixth instrument gets named is decided when budget is actually allocated, not before.

The bench

Backup options — evaluated, held in reserve

Neither of these is in the current Stack, and neither has been rejected. They're evaluated, understood, and ready if a seat opens — which is exactly what failsafe Scenario 1 assumes exists.

ComfyUI — the standing backup for the visual seat

Self-hosted, node-based image and video generation. Not in the Stack today because it means running GPU compute and building workflows by hand — more setup and maintenance than a two-person team needs while Higgsfield covers the seat. Where it earns its bench place: if Higgsfield's pricing or API ever made the seat unviable (Scenario 1), the same city-tagged visual brief — subject, mood, reference asset, brand cues — feeds a ComfyUI workflow instead. The brief format is tool-agnostic on purpose; the switch would cost setup time, not a redesign. It also brings capabilities Higgsfield doesn't lead on — style transfer, upscaling — so it could eventually join as a complement rather than only a replacement.

Sentry — error monitoring, for when a pipeline actually runs unattended

Automatic error capture with enough context that Claude could read a failure and retry or fix it. Not adopted yet for a plain reason: there's no unattended pipeline to monitor — at Level 1 (Claude in a session), a human sees every error as it happens. The moment a Level 2 or 3 scripted pipeline goes live, this moves from bench to requirement, because at that point errors happen with nobody watching. The one adaptation needed: its usual alerting assumes Slack, so alerts would route to email or a queue check instead, matching the tools actually in use.
